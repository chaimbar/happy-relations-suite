## שלב 0 — MCP-First Protocol (חובה לפני כל עבודה על סינריו קיים)

לפני שמשתמשים ב-Playwright או מבקשים מחיים לייצא Blueprint:

### 1. חלץ את ה-Scenario ID מהקישור
קישור לדוגמה: `https://us1.make.com/265818/scenarios/4755722/edit`
→ ה-ID הוא **4755722**

### 2. קרא את ה-Blueprint ישירות דרך MCP
```
mcp__a8e44e87-80d9-41c7-af84-53fef2d18cd6__scenarios_get(scenarioId: 4755722)
```
→ מחזיר Blueprint JSON מלא: מודולים, mappings, filters, scheduling, connections.

### 3. אם התוצאה גדולה (>90K תווים)
שלח לסוכן עם הוראות ניתוח ספציפיות:
- רשימת כל המודולים + URLs
- מבנה זרימה (מה מחובר למה)
- פילטרים ו-Routers
- מודולים ניתוקים (orphans)

### 4. כלים נוספים זמינים
| Tool | שימוש |
|---|---|
| `scenarios_list` | רשימת כל הסינריוס בחשבון |
| `extract_blueprint_components` | ניתוח connections, hooks, keys נדרשים |
| `validate_blueprint_schema` | בדיקת תקינות Blueprint לפני ייבוא |
| `connections_list` | רשימת חיבורים קיימים בחשבון |
| `rpc_execute` | הרצת Make API calls ישירות |

### 5. מתי להשתמש ב-Playwright
רק אם:
- צריך screenshot ויזואלי לאבחון בעיה
- MCP מחזיר שגיאה
- צריך לבצע פעולה ידנית בממשק (כגון יצירת connection חדש)

**לעולם אל תנסה login דרך Playwright לפני שבדקת MCP.**

## ידע ליבה — Make.com

### מבנה סינריו
סינריו ב-Make.com הוא שרשרת מודולים. כל מודול עושה דבר אחד. הנתונים זורמים כ-**Bundles** — כל Bundle הוא יחידת מידע אחת (מייל אחד, שורה אחת, אירוע אחד).

### סוגי מודולים
| סוג | מה עושה | דוגמה |
|---|---|---|
| **Trigger (Polling)** | מתעורר לפי לוח זמנים, בודק אם יש נתונים חדשים | Watch Rows בגוגל שיטס |
| **Trigger (Instant/Webhook)** | מתעורר מיד כשמשהו קורה | Webhook מ-Typeform |
| **Action** | CRUD — יוצר, עורך, מוחק, קורא | Add Row, Send Email |
| **Search** | מחפש רשומות לפי קריטריונים | Search Contacts ב-HubSpot |
| **Router** | מפצל לנתיבים לפי תנאים | אם לקוח > 1000₪ → נתיב A, אחרת → B |
| **Filter** | שומר רק Bundles שעומדים בתנאי | רק מיילים מ-domain מסוים |
| **Iterator** | פורס מערך ל-Bundles נפרדים | 5 קבצים → 5 Bundles |
| **Aggregator** | מאחד Bundles חזרה למערך | 5 URLs → מערך אחד |
| **HTTP Module** | קריאת API חיצוני ללא אינטגרציה | POST ל-webhook מותאם אישית |
| **Data Store** | מסד נתונים פנימי ב-Make | שמירת מצב בין הרצות |

### טיפול בשגיאות
| Handler | מה עושה | מתי להשתמש |
|---|---|---|
| **Rollback** | ברירת מחדל — עוצר הכל, מבטל שינויים | שגיאות קריטיות |
| **Skip** | דלג על ה-Bundle הבעייתי, המשך | נתונים חלקיים שאפשר להתעלם |
| **Resume** | החלף תוצאה כושלת בערך ברירת מחדל | כשיש fallback הגיוני |
| **Retry** | שמור כ-Incomplete Execution לטיפול ידני | שגיאות זמניות (timeout, rate limit) |
| **Commit** | עצור ושמור מה שנעשה עד כה | חשוב לשמור progress חלקי |

### Operations — חיסכון
- כל מודול שרץ = 1 Operation
- Aggregation לפני כתיבה: 100 שורות בנפרד = 100 ops. אחרי Aggregation = 1 op.
- Webhook = 0 ops להאזנה, 1 op להפעלה
- Filter לפני מודולים כבדים חוסך ops

### Scheduling
- **Every N minutes** — בדיקה מחזורית
- **Specific days/times** — הרצה בזמן מוגדר
- **Instant (Webhook)** — מיידי, ללא polling
- הגדרת Max Cycles למניעת timeout ארוך

## שלב 2 — איסוף מידע מחיים

**שאל הכל בבת אחת:**

```
🎯 מה האוטומציה אמורה לעשות? (מה הטריגר, מה הפעולה הסופית)

📥 מה מפעיל את האוטומציה?
   □ משהו קורה בכלי אחר (טופס, מייל, CRM, שיט)
   □ לוח זמנים קבוע (כל X שעות / יום / שבוע)
   □ בקשה ידנית שלי

📤 מה התוצאה הרצויה?
   (לאן הנתונים הולכים, מה נוצר/נשלח/מתעדכן)

🔗 אילו כלים מעורבים? (Make מחובר אליהם כבר?)

⚠️ מה קורה כשמשהו נכשל? (דלג, שלח התראה, עצור הכל)

🔗 יש סינריו קיים? אם כן — שתף קישור או Blueprint JSON
```

## שלב 3 — ניתוח וארכיטקטורה

לפני הכתיבה, בנה מפה מנטלית:

```
[TRIGGER] → [FILTER?] → [ACTION 1] → [ROUTER?]
                                         ├── Route A → [ACTION 2a]
                                         └── Route B → [ACTION 2b]
```

בדוק:
- האם יש מערכים שצריך Iterator?
- האם צריך לאחד תוצאות עם Aggregator?
- כמה Operations צפויים לרוץ?
- האם Data Store נדרש?

## שלב 4 — פלט מלא

### 4א. תיאור מהיר
```
🤖 אוטומציה: [שם]
📋 מה היא עושה: [משפט אחד]
⚡ טריגר: [polling / instant webhook]
🔢 Operations משוערים: [מספר]
```

### 4ב. מבנה מודולים (צעד-אחר-צעד)

עבור כל מודול:
```
מודול N: [שם האפליקציה] — [שם הפעולה]
├── הגדרות:
│   • [שדה]: [ערך / מיפוי]
│   • [שדה]: [ערך / מיפוי]
├── Data Mapping: [מאיפה מגיעים הנתונים]
└── הערה: [למה זה כאן / מה חשוב לשים לב]
```

### 4ג. Filters & Routers
עבור כל Filter/Router:
```
📍 Filter/Router N:
├── תנאי: [שדה] [אופרטור] [ערך]
└── אם לא עומד בתנאי: [דלג / נתיב אחר]
```

### 4ד. טיפול בשגיאות
```
⚠️ Error Handling:
├── מודול X → [Skip / Resume / Retry] כי [סיבה]
└── הגדרות סינריו: consecutive errors = [מספר]
```

### 4ה. הגדרות Scheduling
```
⏰ תזמון:
└── [Every 15 minutes / Daily 08:00 / Instant Webhook]
```

### 4ו. הוראות יישום
רשימת צעדים מספריים לבנות את הסינריו ב-Make.com:
1. צור סינריו חדש / פתח את הקיים
2. הוסף מודול ראשון: ...
3. הגדר חיבור (Connection): ...
4. ...וכן הלאה עד סוף

### 4ז. בדיקה (Test Checklist)
```
✅ לפני הפעלה:
□ הרץ Run Once ובדוק את הפלט של כל מודול
□ בדוק שה-Data Mapping מחובר נכון
□ וודא שה-Filter לא חוסם הכל
□ בדוק מה קורה כשמגיע Bundle ריק
□ הפעל את הסינריו ב-ON ובדוק שעובד לפי תזמון
```
