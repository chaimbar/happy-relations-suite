# סקיל: ניהול זרימת עבודה

## הפעלה
כשהמשתמש מריץ `/workflow` או אומר:
"תנהל את הפרויקט" / "תכנן את העבודה" / "תעביר לעובד הנכון" /
"אני צריך [משימה] — תדאג שזה יקרה" / "תפעיל את הצוות על [נושא]"

---

## שלב 0 — טעינת צוות (תמיד ראשון)

קרא: `.claude/עובדים/ROSTER.md`

זהה את כל העובדים הפעילים. לכל עובד — קרא את ה-`profile.md` שלו כדי להבין:
- מה הוא עושה
- מה הוא **לא** עושה
- מה הסטנדרט שלו (שומר ל / בדיקה עצמית)

**אם נוסף עובד חדש ל-ROSTER מאז הפעם האחרונה** — קרא גם את הסקיל שלו.

---

## שלב 1 — קבלת המשימה

שאל את חיים (הכל בבת אחת):
```
מה צריך לקרות? (תיאור קצר):
יש דדליין / דחיפות?
יש לקוח / פרויקט ספציפי מעורב?
יש תוצר סופי מוגדר (מסמך / הצעה / דוח / קוד)?
```

---

## שלב 2 — תכנון הזרימה

פרק את המשימה לתתי-משימות. לכל תת-משימה:
- **מי:** העובד הנכון (לפי scope ב-profile שלו)
- **מה בדיוק:** תיאור חד-משמעי של הפלט הצפוי
- **תלויות:** האם צריך לחכות לפלט מהעובד הקודם?

הצג לחיים לאישור:
```
📋 תוכנית זרימה — [שם משימה]

שלב 1 → [עובד] — [מה עושה] → פלט: [מה יוצא]
שלב 2 → [עובד] — [מה עושה] → פלט: [מה יוצא]
...

האם לאשר ולהתחיל?
```

אם חיים מבקש שינויים — עדכן ושאל שוב.

---

## שלב 3 — לולאת ביצוע (חזור על כל שלב בתור)

### א. בריף לעובד
לפני שהעובד מתחיל, כתוב לו בריף ברור:

```
📨 בריף ל[שם עובד] | שלב [N] מתוך [סה"כ]

המשימה שלך:
[תיאור מדויק]

הקשר מהשלב הקודם:
[מה עשה העובד הקודם + מה יצא — אם רלוונטי]

פלט נדרש:
[מה בדיוק צריך לצאת, לאן לשמור]

גבולות:
[מה לא לעשות / מה מחוץ לתחום]
```

### ב. עובד מבצע
קרא: `profile.md` + `skills/[skill].md` של העובד
הפעל את הסקיל לפי ההוראות שלו.

### ג. בדיקת איכות (QA Gate)
אחרי שהעובד סיים — בדוק את הפלט:

1. קרא את סעיף **"בדיקה עצמית"** מהסקיל של העובד
2. עבור על כל פריט ✅/❌
3. בדוק גם:
   - [ ] הפלט ספציפי ומלא? (לא "כמעט גמור")
   - [ ] נשמר לנתיב הנכון (תוצרים מוגמרים/)?
   - [ ] תואם לבריף שניתן בשלב א׳?
   - [ ] אין שאלות פתוחות שנשארו ללא מענה?

**אם כל הבדיקות עברו:**
```
✅ QA עבר | [שם עובד] — שלב [N]
עובר לשלב [N+1] → [שם עובד הבא]
```

**אם יש בעיות:**
```
❌ QA נכשל | [שם עובד] — שלב [N]

בעיות שנמצאו:
- [בעיה ספציפית 1]
- [בעיה ספציפית 2]

[שם עובד] — תקן את הנקודות הבאות ואחזור לבדוק:
[הוראות ספציפיות לתיקון]
```

חזור לשלב ב' עד שהQA עובר. **לא מוסרים פלט שלא עבר QA.**

## שלב 4 — דוח מסכם

כשכל השלבים הושלמו:

```
# ✅ זרימת עבודה הושלמה — [שם משימה]
📅 [תאריך] | [N] שלבים | [N] עובדים

## מה נעשה
| שלב | עובד | פלט | QA |
|-----|------|-----|-----|
| 1 | [שם] | [תיאור] | ✅ |
| 2 | [שם] | [תיאור] | ✅ |

## תוצרים סופיים
[רשימת קבצים שנשמרו + נתיבים]

## הערות לחיים
[משהו חשוב שעלה תוך כדי? החלטה שצריך לאשר?]
```

שמור ל: `תוצרים מוגמרים/דוחות/workflow-[YYYY-MM-DD].md`

## עדכון אוטומטי לעובדים חדשים

כשמוסיפים עובד חדש דרך `/new-employee`:
1. ROSTER.md מתעדכן אוטומטית
2. דן קורא ROSTER בתחילת כל `/workflow` — כך הוא מכיר את העובד החדש
3. **אין צורך בהגדרה ידנית.** הלולאה עובדת.

## למידה בזמן אמת

זיהוי: אם במהלך הביצוע המשתמש אמר:
**"שנה X"** / **"לא ככה"** / **"פעם הבאה..."** / **"יהיה יותר טוב אם"** / **"תזכור ש..."**

**כשמזהה הערה — בסוף המשימה, לפני הבדיקה העצמית:**
> "שמתי לב שהערת: *[ציטוט]*. האם לעדכן את הסקיל שלי לפי זה?"

אם כן → הפעל `/improve` עם ההקשר:
`"הערת שיפור ב-workflow של דן: [ציטוט]. [שלב רלוונטי אם ידוע]."`

ניר יציג before/after לאישורך — ורק אז מחיל.
